# PropFlow V3.1 — Productionising brief

Deployment target: Vercel + Supabase free tier. 
MVP Target scale: ~100 customers.

---

## Current State

V3 is complete. The app is functional and tested at a surface level but
was built quickly as an MVP without explicit conventions, formal code
review, or production quality standards. The codebase has not been
audited. The goal of V3.1 is to bring it to production quality before
the visual redesign (V4: UX/UI refresh + shadcn migration) and new features (V5+ e.g. insights & trends).

---

## Workstreams (in order)

```
1. Conventions & standards     — naming, structure, API patterns, Drizzle
                                  usage, type safety rules. Produces
                                  docs/conventions.md.

2. Quality gates & CI/CD       — TypeScript strict, ESLint, GitHub Actions. 
                                  Produces docs/quality-gates.md and CI config.

3. Observability               — Structured logging (lib/logger.ts),
                                  consistent error handling across all
                                  API routes, Sentry integration for
                                  error alerting. Produces
                                  docs/observability.md.

4. Codebase audit              — Targeted audit of the 4 high-risk areas
                                  that lint/tsc cannot catch:
                                  (1) RLS policy coverage across all tables,
                                  (2) soft-delete WHERE clause completeness,
                                  (3) financial calculation correctness
                                      (yield, LVR, cash flow aggregations),
                                  (4) auth check completeness on API routes.
                                  Document findings by severity: must fix
                                  before release / should fix / nice to have.
                                  Produces docs/audit.md.

5. Cleanup                     — Work through audit findings in severity
                                  order. Small reviewable slices.
                                  Refactoring only — no behaviour changes.

6. Testing                     — Assess coverage gaps. Ensure critical
                                  paths, financial calculations, and RLS
                                  isolation are explicitly tested. Produces
                                  docs/testing-strategy.md.

7. Deploy                      — Stand up production Vercel + Supabase.
                                  Run migrations against production DB.
                                  Verify end-to-end in production.
                                  Produces docs/deployment.md.
```

Deploy is the final milestone of V3.1 — the codebase is in its cleanest
state, conventions are enforced, quality gates are passing, and the
production migration story is fully understood before the first deploy.

---

## Quality Bar

Build for correctness, auditability, and maintainability — not
scalability. Target: ~100 customers on Vercel + Supabase free tier.

The risks worth protecting against:

- Wrong financial calculations (yield, LVR, cash flow aggregations)
- Data loss or mutation bugs (soft deletes, RLS gaps)
- A future change (by Claude Code or by hand) breaking something
  non-obviously

Not a concern at this scale: horizontal scaling, caching strategy,
connection pooling beyond Supabase defaults.

## Document Strategy (Context Management)

Claude Code sessions have a finite context window. All decisions and
conventions must be written to committed markdown files — these are the
persistent memory that survives context resets.

### Reference documents (live in /docs)

```
docs/conventions.md          Code standards, naming, file structure,
                              API patterns, Drizzle query patterns,
                              type safety rules. Claude Code reads this
                              at the start of every session.

docs/observability.md        Logging conventions, error handling
                              pattern, Sentry setup.

docs/audit.md                Findings by severity. Input to cleanup.
                              Produced by Claude Code after reading
                              the codebase.

docs/testing-strategy.md     What must be tested and why. Critical
                              paths listed explicitly.

docs/deployment.md           Vercel + Supabase production setup.
                              Migration strategy (not db reset).
                              Environment variables. CI/CD pipeline.
```

### Session handoff pattern

At the end of every session, or before the context window fills:

1. Write all decisions made to the relevant doc
2. Update the session log with what was completed and what is next
3. The next session starts by reading: this brief + the relevant
   workstream doc + the session log

### Session log — docs/v3.1-session-log.md

A running log of progress across all sessions. Format:

```
## Session N — [date]

Workstream: [which one]

Completed:
- [what was done]

Decisions made:
- [any convention or design decision, with rationale]

Next session should:
- [concrete first task]
- [docs to read before starting]
```

This log is the single source of truth for where we are. Update it
before ending every session, even if the session was short.

## Key Decisions Already Made

These are settled — do not re-open them:

- **Live ledger queries** — no stored totals. All financial numbers
  computed from property_ledger_entries at query time.
- **Soft deletes** — deleted_at column on ledger entries and source
  documents. All queries must include WHERE deleted_at IS NULL.
- **Commentary only in portfolio_reports** — totals and flags columns
  dropped in V3. Table stores aiCommentary, version, timestamps only.
- **Feature flags** — lib/flags.ts with aiCommentary: false. Commentary
  generation is behind this flag.
- **No entity ownership percentages** — entityId on properties and
  loan_accounts is a simple FK. Junction tables deferred to V5+.
- **loan_payment entries require loanAccountId** — enforced at API
  layer, not DB constraint.
