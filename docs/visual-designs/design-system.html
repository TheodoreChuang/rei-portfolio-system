12 — Interaction patterns

How Folio behaves, not just how it looks.

Decisions on editing and confirming that should be the same everywhere in the app. One edit pattern, two confirmation tiers — chosen for a calm, deliberate, single-user product. Each entry below is a decision, not a suggestion.

Rule of one
One edit pattern across the app. Don't mix inline-edit with edit-modes with drawers.
Stakes > speed
If an action takes input, cascades, or can't be reversed cheaply — make the user pause.
Reconsider when
Folio adds bulk operations or a high-velocity triage flow. Until then, YAGNI.
12a · Editing

Inline edit-in-place. One pattern, applied everywhere.

Every editable value reveals an affordance on hover and becomes an input on click. No edit modes, no per-record drawers, no "Save / Cancel" footers.
Core entities (properties, loans, entities) don't change often. When they do, the user is making a single, considered tweak — not filling a form. Inline edit keeps the record visible during the edit and removes a whole layer of UI chrome. Relationship fields (entity, agent) use inline pickers; lifecycle actions (sold, archive, delete) live in the overflow menu and use confirmations — they are not edits.

Use inline edit
  • Single field on a known record (nickname, weekly rent, lease date).
  • Relationship pick (entity, agent) — opens an inline picker, not a modal.
  • Anywhere the user reads the value and the edit; both should stay visible.
Do not use inline edit
  • Creating a record (e.g. adding a property) — that's a dedicated flow.
  • State transitions (mark as sold, archive, delete) — use confirmations.
  • Multi-field forms with cross-validation — rare, but reach for a modal.
Affordance
Hover reveals a subtle surface-sunken background and a small pencil glyph. Editable fields never need a "Click to edit" label — the affordance is the contract. If a field can't display its affordance clearly, the field is not editable in place.
12b · Confirming

Two tiers. Inline for contextual, modal for explicit.

If an action takes input → modal. If it's a single contextual yes/no → inline confirm. If it's irreversible or cascades to other records → always modal.
Confirmations are the moment the app slows the user down. We tier them so the friction matches the stakes: contextual deletes and dismissals stay in place, while state transitions and destructive actions become an explicit, focused decision. Folio does not use toast-with-Undo — see the note below.
Action shape Pattern Example
Single contextual yes/no Inline confirm Delete one valuation row · dismiss a prompt
State transition with data Modal Mark property as sold · close a loan
Destructive on a core entity Modal · destructive Delete property · delete entity with linked records
Reversible bulk action — (not adopted yet) Bulk archive · bulk re-categorise

Toast · undo
Deliberately not adopted yet. Folio is single-user, deliberate, and low-velocity — the patterns that earn Toast-with-Undo (Gmail-style triage, multi-user shared state, mobile fat-finger taps) don't apply. Two patterns are cheaper to learn and to build. Revisit when we add bulk operations on transactions.
