## Session 1 — 2026-05-01

Workstream: 1 — Conventions & Standards

Completed:
- Read all significant files: db/schema.ts, all lib/ files, representative API routes
  (properties, entities, upload, reports, ledger/summary), middleware, next.config.ts,
  tsconfig.json, dashboard page, properties detail page, app-nav component
- Discussed all 7 convention areas with user, agreed on each before writing
- Produced docs/conventions.md

Decisions made:
- **Inline queries** stay in route handlers; no service/repository layer at this scale.
  Extract to lib/ only when logic is independently testable or shared.
- **Zod for all request body parsing** in routes (already used for AI output in
  lib/extraction/schema.ts). Eliminates `unknown` narrowing and `as` casts in routes.
- **No `as` casts** in route business logic. SDK type gaps (e.g. Supabase StorageApiError)
  isolated to named utility functions in lib/.
- **PATCH not PUT** for partial updates. properties/[id]/route.ts currently uses PUT —
  cleanup task.
- **lib/env.ts** for centralised env var access with fail-fast validation. No more
  `process.env.X!` scattered across files.
- **No route summary comments** — Zod schemas serve as the API contract/documentation.
  No docs/api.md and no OpenAPI for now; revisit in V5+ if external consumers appear.
- **Index naming** convention is `idx_{table}_{column_or_purpose}`. One outlier
  (`properties_user_id_idx`) to fix at cleanup.
- **No 422 status** — remove the one instance in reports route.
